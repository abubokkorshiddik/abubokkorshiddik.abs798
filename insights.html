Threshold selection in outbreak prediction models

Md. Abu Bokkor Shiddik
Researcher · Statistics, BRUR

Outbreak prediction models generate probability outputs that represent risk of disease occurrence. These probabilities cannot be directly used for decision making — they are converted into binary outcomes using a threshold. This step strongly affects how early we detect outbreaks, how many false alarms we get, and how reliable the system becomes in real public health use.

Table 1. Basic concept of threshold selection

This table explains the basic idea behind threshold selection and its direct interpretation in real systems.

ConceptSimple explanationReal discussion
ThresholdProbability cutoff for outbreak decisionControls sensitivity to detecting early disease signals and shapes decision stability
Low thresholdMore alerts generatedBetter for early warning but increases false alarms in routine surveillance
High thresholdOnly strong signals detectedReduces false alarms but may miss early outbreak signals in weak transmission periods

Overall, threshold acts like a balance control between early detection and operational stability.

Table 2. Suitability levels of threshold methods

Suitability levelWhat it meansDiscussion in real systems
Highly suitableReliable and widely usedUsed in operational outbreak systems where stability is important
Moderately suitableCondition dependentRequires tuning and validation before deployment
Less suitableMainly theoreticalUsed mostly in academic comparison rather than real surveillance systems

Table 3. Highly suitable threshold methods

MethodSimple ideaDiscussionUsefulness
Precision-recall basedBalances detection and precisionBest for imbalanced outbreak data where missing cases is criticalVery high
Cost-sensitive thresholdingMinimizes real world lossUseful when outbreak miss cost is higher than false alarm costVery high
Percentile basedFlags top risk valuesSimple and practical for early warning dashboardsHigh
Mean minus SDDetects abnormal riseUseful in noisy surveillance data with weak labelingModerate to high
Seasonal adaptiveChanges with seasonVery important for climate-sensitive diseases like dengueVery high
Spatial adaptiveRegion-specific thresholdImproves accuracy in multi-region outbreak monitoring systemsVery high

Table 4. Moderately suitable threshold methods

MethodWhat it doesDiscussion
ROC based (Youden Index)Balances sensitivity and specificityDoes not handle imbalance well in outbreak datasets
Maximum sensitivityDetects most outbreaksLeads to many false alarms in real systems
Maximum specificityReduces false alertsMay delay outbreak detection timing
Z-score thresholdDetects statistical spikesBetter suited for anomaly detection than ML prediction
Rolling thresholdUpdates over timeCan become unstable in small datasets

Table 5. Less suitable methods in operational systems

MethodWhy it is weakDiscussion
0.5 thresholdAssumes balanced dataNot realistic for rare outbreak events
Mean probabilityNo epidemiological meaningDepends heavily on model output distribution
Median thresholdIgnores risk structureOnly splits data without decision logic
Bayesian thresholdNeeds strong priorsDifficult in real surveillance settings
Expected utilityHard to define costsTheoretical but not operationally simple

Final insight

In real outbreak prediction systems, threshold selection is not a fixed rule but a decision framework. The most effective systems combine precision-recall optimization, cost-sensitive learning, and adaptive strategies — seasonal and spatial thresholds — to ensure stable and reliable public health action.

Understanding GATHER and making health estimates transparent

Md. Abu Bokkor Shiddik
Researcher · Statistics, BRUR

In health research, we often produce numbers that directly influence decisions — national rates, trends, and future projections. These estimates guide policies, funding, and intervention strategies. But how transparent are these results? GATHER (Guidelines for Accurate and Transparent Health Estimates Reporting) provides a structured framework to report health estimates clearly and reproducibly.

Why GATHER matters

Health estimates often rely on multiple datasets, statistical or machine learning models, assumptions and adjustments, and uncertainty calculations. Without proper reporting, results may be difficult to interpret and trust. GATHER ensures clarity and reliability across all these dimensions.

GATHER reporting domains

CategoryFocus areaPurpose
Study definitionPopulation, time, geographyDefines scope
Data inputsSources and characteristicsEnsures traceability
Data processingCleaning and adjustmentImproves reproducibility
ModelingMethods and assumptionsExplains estimation
ValidationModel checkingEnsures reliability
OutputsResults and uncertaintyImproves interpretation
TransparencyData and code sharingSupports replication

GATHER checklist (18 items)

NoItemDescription
1PopulationDefine scope and time
2OutcomesSpecify estimates
3FundingDisclose funding
4Data sourcesList datasets used
5Data accessExplain how retrieved
6CriteriaInclusion / exclusion
7CharacteristicsDescribe the data
8ProcessingCleaning steps taken
9AvailabilityShare or restrict
10OverviewConceptual framework
11ModelsStatistical methods used
12JustificationWhy this model was used
13AssumptionsState assumptions clearly
14UncertaintyExplain how calculated
15ValidationModel evaluation approach
16ResultsInclude confidence intervals
17CodeShare if possible
18ReplicationEnsure reproducibility

Example of reporting estimates

IndicatorEstimateUncertainty interval
Disease prevalence12.5%(10.2% – 14.8%)
Mortality rate145 per 100,000(120 – 170)
Projection (2030)18.3%(15.0% – 21.6%)

Practical tips

  • Always report uncertainty alongside point estimates
  • Explain model choice clearly with justification
  • State assumptions explicitly — never hide them
  • Document missing data handling procedures
  • Include validation results for all models

GATHER promotes transparent and trustworthy science. It helps ensure that research findings are clear, reproducible, and useful for real-world decisions — from government resource allocation to global intervention planning.

Normalization vs. Standardization in data preprocessing

Md. Abu Bokkor Shiddik
Researcher · Statistics, BRUR

When preparing data for machine learning models, features often have very different ranges and units. Without scaling, models that rely on distances or gradient-based optimization can perform poorly. Normalization and standardization are two core feature scaling techniques used to address this — but they work differently and suit different scenarios.

What is normalization?

Normalization scales data to a fixed range, usually 0 to 1. The most common form is min-max normalization, where each value is rescaled relative to the minimum and maximum values of the feature.

Xnorm = (X − Xmin) / (Xmax − Xmin)

After this transformation, all values fall in [0, 1]. This is useful when features have unknown distributions or are not Gaussian, and works well with distance-based algorithms like k-NN and K-Means.

What is standardization?

Standardization (z-score scaling) transforms features so they have mean = 0 and standard deviation = 1. This centers the data and scales relative to its spread.

Xstd = (X − μ) / σ

This is especially helpful for gradient-based models — SVM, Logistic Regression, Neural Networks — and for techniques like PCA that are sensitive to feature variance.

Key differences

FeatureNormalizationStandardization
Scaling goalFixed range (0 to 1)Mean = 0, Std = 1
Typical range[0, 1] or [−1, 1]Unbounded
Formula(X − Xmin) / (Xmax − Xmin)(X − μ) / σ
Best forDistance-based modelsStatistical & ML models
Outlier sensitivityHighLower

When to use which

Use normalization when: data does not follow normal distribution, using distance-based algorithms, or bounded values are required.

Use standardization when: data is approximately Gaussian, using regression, SVM, or PCA, or the model depends on variance.

Example in Python

from sklearn.preprocessing import MinMaxScaler, StandardScaler

# Normalization
norm_scaler = MinMaxScaler()
X_norm = norm_scaler.fit_transform(X_train)

# Standardization
std_scaler = StandardScaler()
X_std = std_scaler.fit_transform(X_train)

Practical tips

  • Normalization is sensitive to outliers — handle them first
  • Standardization is more stable when outliers are present
  • Always fit the scaler on training data only
  • Apply the same transformation to test and validation data

In ML literature, "normalization" sometimes refers to standardization — the terms are used loosely. But in scikit-learn practice, normalization = MinMaxScaler, standardization = StandardScaler.
